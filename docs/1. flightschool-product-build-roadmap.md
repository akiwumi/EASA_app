## 2. Product Goal

Turn the current EASA regulation update app into a flight-school platform that helps an ATO or pilot training organisation:

- manage manuals and revisions
- monitor EASA changes
- map regulatory changes to training manuals and SOPs
- assign reading to students and instructors
- track acknowledgements and sign-offs
- manage training programmes, phases, lessons, and attached documents
- keep a full audit trail and rollback history
- present a polished landing page for sales and demos

## 9. Product Principles

Cursor should follow these product rules while building:

- do not remove existing working screens unless replacing them with better versions
- reuse existing dashboard, updates, history, flightbooks, and admin foundations
- keep manual approval at the center for compliance-sensitive changes
- separate student, instructor, and admin UI clearly
- make the mobile experience clean and simple for students
- prefer simple workflows before advanced automation

## 10. Build Phases

Build in this order.

## Phase 1: Stabilise and reorganise

### Goal

Make the current app easier to extend without breaking working features.

### Cursor tasks

- audit current routes, components, API routes, and Supabase queries
- create a route map document in code comments or a local docs file if helpful
- make shared types for organizations, flightbooks, updates, training entities
- clean up duplicated UI helpers
- keep existing features working

### Done when

- current app still builds
- no existing route is broken
- shared data types are easier to extend

## Phase 2: Landing page upgrade

### Goal

Turn the homepage into a real product landing page for flight schools.

### Sections to build

- hero
- problem section
- value grid
- role-based section
- product workflow section
- feature proof section
- final CTA

### Page messaging

- this is a training and compliance platform for flight schools
- it connects manuals, training reading, acknowledgements, and EASA awareness
- it is not only a regulation monitoring tool

### Done when

- homepage clearly explains the product
- a flight school owner can understand the offer in under 20 seconds

## Phase 3: Data model upgrade

### Goal

Add training and school operations tables without breaking current compliance features.

### Cursor tasks

- create new SQL migration files for training programmes, phases, lessons, assignments, acknowledgements, sign-offs, forms, branding, and onboarding
- add indexes and RLS
- create views for dashboard summaries

### Manual tasks

- run the SQL in Supabase SQL Editor
- verify tables appear
- verify RLS is enabled

### Done when

- new tables exist
- old features still work
- data can be queried by organization safely

## Phase 4: Training module build

### Goal

Add the first truly differentiated flight-school workflows.

### Core features

- training programmes list
- programme detail page
- phases within a programme
- lessons within a phase
- attach manuals and documents to lessons
- assign reading to students and instructors
- collect acknowledgements
- record instructor sign-offs

### Done when

- a school admin can create a programme, phase, and lesson
- a lesson can have attached reading
- a student can acknowledge a reading assignment
- an instructor can sign off a training item

## Phase 5: Flightbook and manual library upgrade

### Goal

Make manuals easier to manage and use inside training workflows.

### Cursor tasks

- improve the flightbook browser
- add better metadata and tagging
- show linked lessons and assignments for a manual section
- add section-level comments if practical
- make manual import flow clearer

### Done when

- a manual can be used both for compliance and training delivery

## Phase 6: Dashboard upgrade

### Goal

Make the dashboard useful to school admins, instructors, and compliance teams.

### New dashboard cards

- unread critical updates
- students pending acknowledgement
- instructors pending sign-off
- lessons affected by recent regulatory changes
- latest pipeline run health
- newest proposed updates

### Done when

- dashboard is useful for daily operations, not just technical status

## Phase 7: Notifications and role-specific UX

### Goal

Improve clarity for each user type.

### Cursor tasks

- student notification center
- instructor workload widgets
- compliance manager action queue
- role-based empty states and copy

### Done when

- each role sees a tailored experience

## Phase 8: Search and grounded AI

### Goal

Use the RAG foundation already in the app to improve discoverability.

### Cursor tasks

- search across manuals and approved updates
- filter by programme, phase, document type, and manual
- add AI answers grounded only in stored content
- show provenance and source links

### Done when

- users can find the right section quickly
- AI answers cite the underlying stored manual or regulation sections

## Phase 9: Exports and operational polish

### Goal

Finish the admin and compliance experience.

### Cursor tasks

- exports for manual versions and training records
- onboarding checklist UI
- branding settings page
- pricing page and how-it-works page

### Done when

- app looks commercially usable
- admins can onboard a school with less hand-holding

## 13. Supabase Pieces Cursor Must Respect

Cursor should assume:

- RLS is required on every organization-owned table
- all new training tables must have `organization_id`
- queries must be filtered by organization
- auth users are linked through `org_users`
- approval and rollback features already exist and should be reused

## 17. Acceptance Criteria

The upgrade is successful when:

- the app still supports current regulation-monitoring workflows
- the landing page sells the product clearly to flight schools
- training programmes, phases, lessons, assignments, acknowledgements, and sign-offs exist
- manuals can be used in both compliance and training workflows
- dashboards show operational school metrics
- Supabase remains secure through RLS
- a novice can run the setup by following the SQL instructions file

## 18. Final Advice

Move in phases.

Do not try to perfect the whole product at once.

The best early milestone is:

- strong landing page
- stable schema upgrade
- training programmes
- lesson-linked reading
- student acknowledgements
- instructor sign-offs

That is the point where EASA_app stops looking like a regulation utility and starts looking like a real flight-school platform.
